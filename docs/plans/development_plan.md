# Комплексный План Разработки: Omnichannel CRM & Loyalty (ERPNext)

*Таймлайн: 3 месяца. База: 150k -> 500k пользователей. Бюджет: 500 000 ₽ (CAPEX).*

## 1. Обзор Проекта

### 1.1 Бизнес-Цель
Создание легковесной, экономичной CRM-системы (или шлюза над open-source CRM) для кофейни для замены дорогих сервисов коммуникации и лояльности, сокращающей ежемесячные расходы (OPEX) до уровня 50 000 - 100 000 ₽. Фокус системы — обеспечение работы программы лояльности (начисление/списание бонусов, проверка баланса) и высоконагруженные рассылки (триггерные + массовые акционные).

### 1.2 Каналы (Омниканальность)
Интерфейс системы должен предоставлять одинаковые возможности лояльности через:
- **Telegram**
- **WhatsApp** 
- **ВКонтакте**
- **Mobile App** (интеграция по API)

### 1.3 Ключевые Требования законодательства (152-ФЗ)
- Явный сбор и логирование согласий.
- Серверы физически в РФ.
- Отзыв согласия в 1 клик.
- Минимизация хранимых персональных данных (только телефон, имя, история транзакций).

## 2. Архитектура Системы (High-Load Oriented)

В связи с размером базы (150к сейчас, до 500к через год) и большим объемом рассылок, архитектура должна быть построена на очередях и асинхронных процессах, иначе рассылки заблокируют работу API по начислению лояльности. Рассматривается использование **готового Open-Source решения** (типа ERPNext/Frappe) для ускорения сроков и удешевления MVP.

```
┌─────────────────┐       ┌─────────────────┐       ┌─────────────────┐
│ Омниканальность │       │   Middleware    │       │     ERPNext     │
│ (ТГ, ВК, WA)    │◄─────►│    (FastAPI)    │◄─────►│   (Master DB)   │
└─────────────────┘       └─────────────────┘       └─────────────────┘
        ▲                          │
        │                          ▼
┌─────────────────┐       ┌─────────────────┐
│   Mobile API    │       │   Celery / MQ   │ ──(Пакетная обработка рассылок)
└─────────────────┘       └─────────────────┘
```

## 3. Этапы Разработки (Timeline = 3 месяца)

### Месяц 1: Архитектура, Безопасность и Интеграции
**Цель:** Подготовить стабильное "ядро" лояльности и 152-ФЗ комплаенс.
- Настройка базового сервера (РФ). Развертывание PostgreSQL, Open-source CRM (ERPNext) и Middleware (FastAPI).
- Реализация механики `152-ФЗ` (сбор согласий, структура БД, кнопка "Удалить профиль").
- Интеграция с Telegram API, WhatsApp Business API (если нет ограничений) и VK API для базового отклика (команды типа `/start` и чекбокс согласия).

### Месяц 2: Лояльность и API-паритет
**Цель:** Настроить ядро программы лояльности для всех каналов.
- Внедрение бизнес-логики: проверка баланса, начисление и списание бонусных баллов.
- API для Мобильного приложения (обеспечение паритета: те же функции, что и в ТГ).
- Адаптация ERPNext Custom Fields для хранения баланса клиентов, привязанного к их мессенджер-ID или номеру телефона.

### Месяц 3: High-Load Рассылки и Запуск MVP
**Цель:** Инструментарий оповещения 150 000+ клиентов.
- Подключение очередей сообщений (Celery/Redis/RabbitMQ) для фоновой пакетной отправки сообщений (Batch processing).
- Троттлинг рассылок (Rate Limiting), чтобы ВКонтакте, Telegram и WhatsApp не блокировали ботов за спам (например, не более 30 sms/сек).
- Внедрение триггеров: "Welcome-бонусы" и акционные push-информирования.
- Оптимизация системы под CAPEX лимиты (500k) и запуск проекта.

## 4. Стратегия Масштабирования (Месяцы 4-12)
После успешного достижения MVP в 3 мес:
- **Инфраструктура:** Мониторинг производительности БД на объеме до 500k клиентов. Возможная интеграция Read-Replica узлов баз данных. Облачные технологии в рамках OPEX лимита.
- **Маркетинг:** Сложные цепочки писем, A/B-тесты для акционных рассылок.
- **Сегментация:** Автоматическое тегирование пользователей в ERPNext на основе их частоты визитов в кофейни (VIP, Спящие, Новички) и авто-рассылка специфичных промо-акций.

## 5. Бюджетирование и Выбор Стека
Для укладывания в жесткий **CAPEX в 500 000 ₽**, команда обязуется избегать лишнего создания велосипедов:
- Если стандартный Frappe/ERPNext модуль решает задачу лояльности — используется он (без кастомного питона).
- Если кастомный FastAPI сервис маршрутизирует сообщения быстрее и требует меньше RAM — выносим роутинг в него.
- Мобильная коммерция (возможность оформления заказа-корзины в боте) **ислючается из MVP**, так как 70% времени будет уходить на сложные интеграции с корзиной, отнимая фокус у главного — Триггерных рассылок и работы с бонусами (суть ТЗ клиента).
