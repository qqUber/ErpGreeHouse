# План стабилизации и тестирования ErpGreeHouse

## 1. Фиксация состояния системы

### Чек-лист готовности (Pre-flight Checklist)
Перед запуском любых автотестов необходимо убедиться в следующем:

- [ ] **Инфраструктура**
  - [ ] Redis запущен и доступен (порт 6379/tcp)
  - [ ] База данных SQLite доступна и имеет корректную схему
  - [ ] Backend (FastAPI) запущен и отвечает на healthcheck (порт 8000)
  - [ ] Frontend (Vite) собран или запущен в dev-режиме (порт 5173)

- [ ] **Данные**
  - [ ] Наличие обязательных системных пользователей (admin, operator, etc.)
  - [ ] Отсутствие "мусорных" данных от предыдущих упавших тестов (чистый стейт)
  - [ ] Валидность токенов и сессий (отсутствие просроченных кук)

### Инструментарий
Для автоматизации проверок используются скрипты в `middleware/scripts/`:
- `healthcheck.py`: Проверка доступности сервисов.
- `snapshot_state.py`: Снятие метрик состояния БД (кол-во записей) для сравнения до/после.

## 2. Матрица покрытия автотестов (Smoke & Critical)

| Тест | Файл | Зависимости | Критерии успеха | Критичность |
|------|------|-------------|-----------------|-------------|
| **Login Flow** | `smoke.spec.ts` | Backend, Redis, DB (users) | Успешный вход, редирект, наличие куки | **BLOCKER** |
| **RBAC Roles** | `roles.spec.ts` | Backend, DB (roles) | Отображение корректной роли в UI | **CRITICAL** |
| **Auth Rejection** | `smoke.spec.ts` | Backend | Сообщение "Доступ запрещён" (401) | HIGH |

## 3. Стратегия тестирования

### Этап 1: Backend & DB (Sanity)
*Цель:* Убедиться, что API работает корректно изолированно от UI.
*Инструменты:* `pytest`, `curl`, кастомные скрипты.
*Критерии:* Все эндпоинты возвращают ожидаемые статус-коды (200, 401, 403).

### Этап 2: Integration (Frontend + Backend)
*Цель:* Проверка связки через API-клиент фронтенда.
*Инструменты:* Ручные проверки / Unit-тесты компонентов.
*Критерии:* Корректная обработка ошибок (проброс сообщений с бэкенда).

### Этап 3: E2E (Playwright)
*Цель:* Полная симуляция действий пользователя.
*Правило:* Запуск ТОЛЬКО после успешного прохождения Этапа 1.

## 5. Принципы разработки (Rules)

1. **Stop the Line:** Если тест падает, новые фичи не пилятся, пока тест не починен.
2. **Fix First:** Сначала исправляем причину падения (баг или тест), потом перезапускаем.
3. **Evidence:** Любое исправление должно быть проверено локально перед пушем.
