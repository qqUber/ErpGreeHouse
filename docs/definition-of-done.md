# Definition of Done (обязательное завершение задачи)

Любая задача считается завершённой только после прохождения всех этапов ниже и фиксации результатов в системе трекинга.

## 1) Проверка компиляции и сборки

### Обязательные проверки

- Полная сборка проекта без ошибок компиляции.
- Проверка артефактов сборки (что действительно сформировалось и соответствует ожидаемой версии).
- Проверка корректности версий и зависимостей.

### Артефакты в этом репозитории

- Admin UI: `admin-ui/dist/` (после `npm run build`).
- Middleware: Docker image из `middleware/Dockerfile` (после `docker build`).
- Конфигурации: актуальные `.env.example`, docker-compose файлы, документы миграций (если добавлялись).

## 2) Полное тестирование

### Обязательные проверки

- Прогон всех автоматизированных тестов: unit + integration + e2e.
- 100% успешное прохождение существующих тестов.
- Покрытие тестами: минимум 80% для затронутых критических модулей (security/auth/сессии).
- Отсутствие отладочной информации в production коде: `System.out.println`, `console.log`, `debugger`.
- Использовать только структурированные логи и систему трекинга для статусов.

### Рекомендуемые команды

- Middleware (полный набор): `middleware/run_tests.sh` или `middleware/run_tests.ps1 -TestType all`.
- Coverage для security/auth модулей (минимум 80%):
  - `pytest --cov=app.admin_auth_api --cov=app.security --cov=app.rate_limit --cov=app.request_context --cov=app.runtime --cov-fail-under=80`
- Pre-commit проверки: `pre-commit run --all-files`.

## 3) UI и функциональное тестирование

### Обязательные проверки

- Поднять полный стек в окружении максимально приближенном к production (локально/staging).
- Пройти все затронутые пользовательские сценарии (позитивные/негативные).
- Проверить в нескольких браузерах/разрешениях и на мобильных устройствах (если релевантно).
- Проверить производительность и отказоустойчивость для ключевых операций.
- Подтвердить корректность интеграций с внешними системами.

### Минимальный набор сценариев (авторизация)

- Вход по паролю: успешный, неверный пароль, отмена процесса.
- Восстановление: неверный код восстановления, успешный сброс, отмена процесса.
- Смена пароля: успешная, неверный текущий пароль, отмена процесса.
- Логаут: сессия очищается, повторный доступ запрещён.
- Проверка отсутствия секретов в браузере: Console/Network/localStorage.

## 4) Документирование результата в трекинге

К задаче обязательно прикладываются:

- Логи/вывод сборки (build logs).
- Отчёты тестирования (pytest html/coverage summary, e2e отчёты если есть).
- Скриншоты/скринкасты UI-проверок (затронутые сценарии).
- Краткая сводка: что проверено, на каком окружении, чем подтверждено.

### Шаблон комментария для задачи

- Build: OK (артефакты: …)
- Tests: OK (unit/integration/e2e)
- Coverage (security/auth): XX% (порог 80%)
- UI: OK (сценарии: …, браузеры/устройства: …)
- Security: OK (CSP/headers, masking, rate limit)
- Attachments: build logs, test reports, screenshots
