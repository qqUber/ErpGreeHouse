# Pre-commit Code Review Checklist

## Общие правила
- Все изменения в документацию вносятся только в dev-ветку через pull-request
- Каждый коммит должен проходить все проверки pre-commit хуков
- Запрещены скрипты, не несущие производственной ценности

## Чек-лист проверки кода

### 1. Структура проекта
- [ ] Отсутствуют дублирующиеся файлы
- [ ] Нет временных файлов и артефактов
- [ ] Правильная структура директорий middleware/
- [ ] Все Python файлы в middleware/app/
- [ ] Тесты в middleware/tests/

### 2. Качество кода Python
- [ ] Black форматирование пройдено
- [ ] isort сортировка импортов выполнена
- [ ] Flake8 линтинг без ошибок
- [ ] Bandit security scan пройден
- [ ] Нет debug statements в продакшен коде

### 3. Архитектура
- [ ] Async/await используется корректно
- [ ] Нет блокирующих операций в async функциях
- [ ] Правильное разделение ответственности
- [ ] ERPNext клиент использует mock mode для тестов

### 4. Тестирование
- [ ] Unit тесты покрывают критичные функции
- [ ] Integration тесты для ERPNext API
- [ ] E2E тесты для основных сценариев
- [ ] Все тесты проходят в CI/CD

### 5. Безопасность
- [ ] Нет хардкоденных секретов
- [ ] Правильная обработка персональных данных (152-ФЗ)
- [ ] Rate limiting настроен
- [ ] Валидация входных данных

### 6. Производительность
- [ ] Redis используется для кэширования
- [ ] Celery для тяжелых операций
- [ ] Оптимизированные запросы к ERPNext
- [ ] Пулл соединений настроен

### 7. Логирование и мониторинг
- [ ] Structured logging используется
- [ ] Правильные уровни логирования
- [ ] Метрики собираются
- [ ] Ошибки обрабатываются корректно

### 8. Документация
- [ ] API документация обновлена
- [ ] README актуален
- [ ] Комментарии в коде при необходимости
- [ ] CHANGELOG обновлен

## Процесс коммита

1. Проверить чек-лист выше
2. Запустить pre-commit hooks: `pre-commit run --all-files`
3. Убедиться в отсутствии ошибок
4. Создать pull-request в dev-ветку
5. Пройти code review
6. Merge после аппрува

## Запрещенные практики
- Скрипты проверки дубликатов файлов
- Автоматическая генерация документации без необходимости
- Сложные pre-commit хуки, не несущие ценности
- Хардкоденные пути и конфигурации
- Игнорирование ошибок линтинга
